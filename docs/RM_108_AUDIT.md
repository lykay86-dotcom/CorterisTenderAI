# RM-108.1 — аудит AI-резюме тендера

## Цель RM-108

Показать пользователю краткое, объяснимое резюме тендера. Резюме не принимает
решение об участии и не заменяет детерминированные результаты анализа,
верификации, стоп-факторов и RM-107.

## Существующие компоненты

| Компонент | Назначение | Использование в RM-108 |
|---|---|---|
| `app/ai/structured_analysis.py` | Строгая JSON-схема AI-анализа с полем `summary` | Использовать как будущий безопасный AI-слой, не копировать схему. |
| `app/tenders/full_analysis.py` | Оркестрация документов, текста и анализа | Источник подтверждённых фактов для offline-резюме. |
| `app/tenders/requirement_analysis.py` | Детерминированные требования и риски | Основной источник фактов и доказательств. |
| `app/tenders/collector/verification.py` | Provenance, confidence и конфликты | Показывать достоверность данных, а не скрывать её. |
| `app/tenders/participation_decision.py` | Итоговое решение RM-107 | Резюме обязано отражать решение без его изменения. |
| `app/ui/tender_full_analysis_dialog.py` | Экран результата полного анализа | Точка интеграции будущего UI, без создания параллельного окна. |
| Windows Credential Manager / настройки AI | Секреты и выбор провайдера | Использовать существующий vault; ключи не сохранять в БД, payload или логи. |

## Вывод аудита

Отдельного доменного модуля AI-резюме нет. RM-108.2 создаст минимальную
доменную модель, а RM-108.3 — детерминированный offline-генератор на основе
уже подтверждённых результатов. RM-108.4 может только улучшать формулировку;
AI не вправе добавлять факты или менять подтверждённое решение RM-107.

## Границы первой реализации

- не выполнять сетевой AI-запрос по умолчанию;
- не дублировать анализ требований и стоп-факторы;
- всегда показывать отсутствие или конфликт доказательств;
- обеспечить offline/fallback результат;
- не изменять схему БД до отдельного решения RM-108.5.
