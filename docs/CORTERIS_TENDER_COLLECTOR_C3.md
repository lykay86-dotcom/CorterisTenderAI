# Corteris Tender Collector — C3 Data Core

## Назначение

C3 принимает результаты нескольких провайдеров, нормализует закупки,
объединяет дубли, сохраняет наблюдения в существующий файл
`tender_registry.sqlite3` и регистрирует изменения между запусками.

Сетевой доступ в этом слое отсутствует. Получение данных остаётся задачей
C2/C4-провайдеров.

## Поток данных

```text
AsyncProviderSearchEngine
    -> TenderNormalizer
    -> TenderDeduplicator
    -> CollectorStateRepository
    -> tender_records + collector_* tables
```

## Уровни идентификации

1. официальный номер ЕИС;
2. межплощадочный регистрационный номер;
3. номер процедуры на конкретной площадке;
4. `source + external_id`;
5. `ИНН/заказчик + название + цена + дедлайн`;
6. хеш нормализованного содержания.

Слабые уровни не заменяют сильные идентификаторы, но позволяют объединить
зеркала одной закупки, когда площадки публикуют разные локальные номера.

## Миграция

Версия схемы: `collector_schema_version = 1`.

Новые таблицы создаются в существующей базе реестра:

- `collector_runs`;
- `collector_run_providers`;
- `collector_run_items`;
- `collector_tender_aliases`;
- `collector_tender_sources`;
- `collector_tender_versions`;
- `collector_tender_changes`;
- `collector_checkpoints`.

Миграция повторно запускаемая. Существующие `tender_records`, архивный статус,
оценка релевантности и пользовательское решение не удаляются и не
перезаписываются служебными значениями коллектора.

## История изменений

Сохраняются:

- новая закупка;
- изменение НМЦК;
- продление или сокращение срока подачи;
- изменение статуса;
- отмена или завершение;
- добавление и удаление документов;
- изменение названия, заказчика или описания.

Каждое изменение содержит источник, время обнаружения, старое и новое
значение.

## Checkpoints

`CollectorCheckpoint` хранит курсор, watermark и JSON-состояние отдельно для
каждого провайдера и области поиска. Это основа инкрементальной загрузки ЕИС и
Портала поставщиков Москвы в следующих этапах.

## Ограничения C3

- провайдер ЕИС ещё не переведён на асинхронный C2-клиент;
- реальный источник Портала поставщиков Москвы ещё не подключён;
- физическое объединение двух уже существующих конфликтующих строк реестра
  пока не выполняется автоматически — конфликт алиасов учитывается в summary;
- UI истории изменений будет добавлен после рабочих провайдеров.
