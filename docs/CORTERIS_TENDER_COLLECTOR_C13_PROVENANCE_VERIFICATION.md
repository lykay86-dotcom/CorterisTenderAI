# Collector C13 — Field Provenance & Verification Core

## Назначение

C13 добавляет отдельную стадию проверки критичных полей между
дедупликацией и рейтингом:

```text
Discovery / Enrichment
→ Normalization
→ Deduplication
→ Verification
→ Ranking
→ Persistence
```

Цель стадии — не просто выбрать непустое значение, а сохранить все
кандидаты, источник каждого значения, уровень доверия, конфликт и основание
выбора.

## Приоритет источников

В ядре используется следующий порядок:

1. официальная документация закупки;
2. ЕИС;
3. официальная электронная площадка;
4. официальный API;
5. официальный сайт заказчика;
6. публичная карточка;
7. агрегатор;
8. неизвестный источник.

Числовой уровень доверия сохраняется вместе с каждым кандидатом. Более
слабый источник не может заменить ранее сохранённое официальное значение.

## Критичные поля

Проверяются и сохраняются:

- номер закупки;
- НМЦК;
- срок подачи;
- статус;
- закон;
- заказчик;
- ИНН заказчика;
- площадка;
- обеспечение заявки;
- обеспечение контракта;
- ссылка на документацию.

Провайдер может передать уточнённые сведения по отдельному полю через:

```python
raw_metadata={
    "field_provenance": {
        "price": {
            "source_id": "official_documentation",
            "source_kind": "official_documentation",
            "source_url": "https://.../notice.pdf",
            "retrieved_at": "2026-07-12T12:00:00+03:00",
            "official": True,
            "verified": True,
            "confidence": 1.0,
        }
    }
}
```

Если field-level metadata отсутствует, применяется консервативная политика
по типу источника.

## Конфликты

Система различает:

- конфликт двух официальных источников одинакового приоритета;
- конфликт источников одинакового приоритета;
- официальное значение против менее надёжного;
- обычное расхождение нескольких источников.

Если два источника одинакового высшего приоритета дают разные значения,
конфликт получает `unresolved=true` и требует ручной проверки.

Если ЕИС расходится с агрегатором, выбирается ЕИС, но значение агрегатора не
удаляется и сохраняется в истории конфликта.

## Защита от понижения достоверности

Перед каждым новым запуском `TenderVerificationService` загружает последние
выбранные значения из базы. Поэтому сценарий:

```text
ЕИС → подтверждённая НМЦК 1 500 000
позднее агрегатор → НМЦК 2 400 000
```

не приводит к перезаписи официальной НМЦК. Агрегатор сохраняется как новый
кандидат и конфликтующий источник.

## Схема базы данных

Версия схемы Collector повышена до 3. Добавлены таблицы:

```text
collector_verification_runs
collector_tender_field_values
collector_tender_field_provenance
collector_tender_field_conflicts
collector_tender_verification_state
```

### collector_tender_field_values

Хранит все значения-кандидаты, включая альтернативные и исторические.
Деньги сохраняются строкой внутри JSON, без преобразования в `float`.

### collector_tender_field_provenance

Хранит источник, URL, время получения, official/verified, confidence и
уровень доверия.

### collector_tender_field_conflicts

Хранит все конфликтующие candidate IDs, выбранное значение, тип конфликта,
критичность и необходимость ручной проверки.

### collector_tender_verification_state

Хранит текущий итог проверки тендера: статус, дату, полноту, конфликты и
минимальную достоверность.

## Резервная копия

Перед миграцией установщик запускает SQLite Backup API, а затем выполняет
`PRAGMA quick_check` на копии. Простое копирование открытой WAL-базы не
используется.

## Ограничения этапа

C13 реализует ядро, миграцию и API чтения. Отдельное визуальное отображение
происхождения каждого поля в карточке тендера запланировано на следующий
этап. Текущий интерфейс сборщика показывает новую стадию проверки, но не
подменяет отсутствие фактической live-проверки провайдера.
