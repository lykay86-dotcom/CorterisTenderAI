# RM-141 as-is user journeys

## Method and notation

These journeys describe production behavior at SHA
`8e704cf74c64e2125ace165807d1a33d3937b739`. They were traced through bootstrap, shell, page,
controller, repository, and existing tests with offline fixtures only. `E/L/P/S/X/C` below means
empty/loading/partial-or-stale/success/error/cancelled. A dash means the state is not meaningful or
has no explicit presentation contract; it does not mean that the state was tested successfully.

## Navigation and context observations

- There is no route object, browser-like history, back action, or formal focus-restoration owner.
- Nine sidebar choices look peer-level although five are placeholders and two are filtered views of
  one workflow repository.
- The Dashboard provides useful deep links, but return state and selection preservation are local
  implementation details rather than a cross-page contract.
- Settings, AI, notifications, and analytics each have both an apparent modern-shell destination
  and meaningful behavior elsewhere or not yet implemented.
- The tender workspace contains the broadest production functionality and must remain available
  throughout RM-142/RM-144 migration; removing legacy actions during RM-141 was prohibited.

## Manual recovery assertions for redesign stages

All later journey acceptance must use temporary data and fake/offline adapters. It must assert that
cancel preserves the previous committed record, destructive recovery names the exact target,
selection/deep-link identity survives route changes, and UI text cannot replace deterministic
score, recommendation, or critical stop-factor priority.
